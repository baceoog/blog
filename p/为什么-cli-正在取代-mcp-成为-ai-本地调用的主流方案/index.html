为什么 CLI 正在取代 MCP 成为 AI 本地调用的主流方案？

Mar 30, 2026

最近在滋养自身因考研初试和复试而近乎荒芜的新知识，发现了一个比较有意思的现象，LLM本身其实只是生成了很多很多的文字而已，那么LLM生成的这些文字，是通过什么方式去对本地的文件进行处理的呢，是如何介入本地工作流的呢？

在探讨这一问题时，面临的问题是 如何建立云端模型推理与本地文件系统之间的安全且高效的映射。 目前行业内存在两种主流演进路径：2024 年由 Anthropic 提出的 MCP 协议，以及 2025 年起开发者集体回归的 CLI 接口。本文将从架构设计、资源消耗及本地数据安全三个维度，解析这一技术转向背后的逻辑。

一、 MCP

MCP 全称为Model Context Protocol，是由 Anthropic 推出的模型上下文协议，旨在为 AI 客户端提供一套标准化的工具调用规范，是标准化的抽象层。

1. 设计初衷与技术指标

  • 权限沙盒化（Sandboxing）：通过MCP Server隔离 AI 逻辑与本地系统，仅暴露受限的函数接口。
  • 跨平台抽象：屏蔽底层操作系统（POSIX/Win32）的差异，将文件操作、数据库查询等行为抽象为统一的 JSON-RPC 接口。

2. 架构逻辑

MCP 采用 Client-Server 架构。云端 LLM 作为 RPC Client，本地运行的 MCP Server 作为 Provider。其交互核心在于 JSON-RPC 2.0 协议：

  1. 静态声明：MCP Server 向 云端 LLM 描述其支持的所有工具及其参数 Schema（遵循 JSON Schema 规范）。
  2. 有状态调用：云端 LLM 根据任务目标，生成符合 Schema 的结构化指令。
  3. 结果回传：MCP Server 执行指令后将结果序列化并返回给模型。

3. 瓶颈

  • Context Window 负载：由于 LLM 需要预先理解所有工具的描述和格式，MCP 方案会将大量的系统提示词空间浪费在 Schema 声明上。实测显示，当接入超过 10 个复杂工具时，Schema 描述可能占据 2k-5k 词元，大大压缩了模型处理实际任务的有效上下文空间。
  • 协议适配成本：开发者需额外编写 Server 代码、处理数据序列化及协议握手，增加了系统的复杂度和潜在的失效点。
  • 非原生训练分布：LLM 在预训练阶段虽然接触过 JSON，但针对 MCP 这一特定协议的训练样本密度远低于通用编程代码，导致其生成符合规范的 RPC 指令时幻觉率很高，较难实现每一次都能生成有效的RPC指令。

二、 CLI

CLI 全称为 Command Line Interface，指通过终端命令行与操作系统或应用程序进行文本交互。而在 AI 时代，其定义我个人觉得可以拓宽为“AI 指令分发与审计接口”，由三个步骤组成：

  1. 推理生成（Inference Generation）：云端 LLM 根据用户自然语言意图，检索预训练语料中的代码分布，生成一段符合对应命令行规范的无状态文本指令串。此时，指令仅作为纯文本数据存在，不具备任何系统访问权限。
  2. 人工审计（Human-in-the-loop Auditing）：生成的文本指令在进入本地执行栈之前，必须经过用户或安全过滤器的语义审查。这一步是安全工程中的隔离点，确保 AI 可能生成的危险偏差（如误删、越权）在转化为物理动作前可被拦截。
  3. 受控执行（Controlled Execution）：在确认指令意图与安全合规后，由本地用户触发执行权，将文本指令提交给系统的 Terminal,根据当前用户的Context完成最终的操作。

1. 为什么 CLI 是 AI 的原生语料？

  • 高数据密度：LLM 的预训练语料库（如 GitHub、Stack Overflow、技术文档）中包含海量的 Shell 指令与 CLI 工具用法。模型对 grep、awk、git 等命令的推理能力天然强于任何新造的私有协议。
  • 无 Schema 依赖：CLI 是无状态或基于约定的，AI 无需在每个 Session 开始前预加载数千行的工具说明，仅需根据当前需求生成单行指令，极大地降低了 Prompt Overhead。

2. 确定性与可观测性

  • Human-in-the-loop：不同于 MCP 的静默调用，CLI 方案天然支持“预览-确认-执行”的人工校验工作流。AI 生成的指令对人类用户具备极高的可读性，方便在执行前进行逻辑审计。
  • 原子化操作：CLI 工具（如 sed、find）通常遵循 Unix 哲学，功能单一且强大。AI 通过组合这些原子化工具，能完成比固定 API 更复杂的长链路任务。

三、 从“授予权限”到“提供建议”

当前趋势表明，AI 本地化的核心正在从定义权限的边界向优化指令流转路径演化。

MCP 试图通过标准化的抽象层解决安全问题，但本质上是赋予了 AI 自动触发本地行为的权限，是存在一定的风险的。而 CLI 将 AI 定位于“指令生成器”，将“执行决策权”交还给个人。这种非侵入式的设计更符合安全工程中的最小特权原则。

主流生产力工具，如Obsidian、飞书、微信等现在都有提供功能完备的 CLI 接口，可以调用可以实现用CLI。

  • Obsidian的 CLI 几乎复刻了 GUI 的所有能力。AI 可以通过指令直接进行 Markdown 语义树的操作、元数据的批量检索与覆盖、以及插件链的触发，而无需维护复杂的 JSON-RPC 服务。
  • 最开始推崇MCP的Claude Code，现在在用户使用层面，也往CLI形式进行转变。
  • 最近爆火的openclaw同样也是选择了 CLI 这条路。

AI 场景下的最小特权原则（PoLP）
在安全工程中，最小特权原则（Principle of Least Privilege, PoLP） 要求系统中的每一个模块、进程或用户只能拥有其完成指定任务所必须的最少权限。将此原则应用于 AI 本地化，其核心差异体现在“权限授予”的阶段性上：

1. 静态授权（MCP 模式）
MCP 试图通过 预定义功能集（Defined Functions） 来实现 PoLP。

  • 逻辑：开发者预先审核并授权 AI 调用某些特定的本地接口（如“读取目录”）。
  • 风险：虽然限制了功能种类，但一旦授权，AI 在执行期（Runtime）内对该接口拥有自动调用的权力。若模型产生逻辑偏差或受到提示词注入攻击（Prompt Injection），它仍可能在授权范围内执行非预期的批量操作（如在授权路径下误删大量文件）。

2. 动态确认（CLI 模式）
CLI 方案通过将 “指令生成” 与 “特权执行” 彻底分离，实现了更高级别的 PoLP：

  • 逻辑：AI 本身 不持有任何特权。它的权限等级为“零”，仅具备输出文本字符串的能力。
  • 执行机制：真实的系统特权仅在用户点击“执行”的那一刻，由本地临时授予给该条特定的指令。
  • 安全增益：这建立了一道的安全防线。即使 AI 生成了恶意指令（如 rm -rf /），由于它不具备执行句柄，该指令在未经过人类审计前仅是一串无害的字符。这种 “按需授权、即用即弃” 的模式，是安全工程中 PoLP 原则的最优实践。

四、CLI 应用案例 ———— 知识库批量属性更新

在不需要打开图形界面的情况下，通过 CLI 组合实现：

  1. 任务需求：更新本地 Markdown 笔记中所有包含特定标签文件的元数据。
  2. AI 输出指令：
    fd -e md --exec sed -i 's/status: todo/status: done/g' {}
  3. 优势：该过程无需为 AI 编写专门的“文件修改 Server”，直接利用 fd 和 sed 这类高效的 Unix 原生工具即可完成。

五、 协议向左，接口向右

MCP 并没有消失，它或许将作为一种企业级规范存在于需要多租户隔离、强审计的受控环境中。

然而，对于个人开发者的本地工作流来说，CLI 似乎已经是利用 LLM 生成的文本数据执行任务的一个很完美的答案了，它利用了 LLM 预训练阶段沉淀的编程知识，规避了新的繁琐的协议负担。
AI 时代，可能不再需要为机器造诸如MCP的新语言，通过 CLI，可以让人类已经沉淀了数十年的指令语言在LLM中大放异彩。
